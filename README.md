# wifi-cursor

[![CI](https://github.com/Cospamos/wifi-cursor/actions/workflows/ci.yml/badge.svg)](https://github.com/Cospamos/wifi-cursor/actions/workflows/ci.yml)
[![Release](https://img.shields.io/github/v/release/Cospamos/wifi-cursor)](https://github.com/Cospamos/wifi-cursor/releases/latest)

Один курсор, несколько компьютеров в одной Wi-Fi сети — как с несколькими
мониторами на одном ПК, только вместо HDMI используется Wi-Fi. Курсор
переносится на соседнее устройство, если довести его до края экрана, либо
мгновенно — по горячей клавише.

Никакого сервера: все устройства в пуле равноправны, поддерживают его
существование сообща (gossip-протокол членства + heartbeat), и пул живёт,
пока в нём остаётся хотя бы одно устройство.

Программа ничего не делает в фоне просто так: захват мыши/клавиатуры
включается только после того, как устройство реально вошло в пул (через
`create` или `join`) — до этого момента процесс либо не запущен, либо ждёт
ввода ID, не трогая системный ввод.

## Установка

Готовые бинарники собираются автоматически в GitHub Actions при каждом
релизе — собирать из исходников не обязательно.

### Windows

```powershell
irm https://raw.githubusercontent.com/Cospamos/wifi-cursor/master/install.ps1 -OutFile install.ps1
.\install.ps1
```

Скрипт спросит, куда установить (по умолчанию `%LOCALAPPDATA%\wifi-cursor`)
и добавлять ли утилиту в PATH — ничего руками искать/копировать не нужно.
Параметры можно передать сразу, без вопросов:

```powershell
.\install.ps1 -InstallDir "C:\Tools\wifi-cursor" -AddToPath y
```

### Linux

```bash
curl -fsSL https://raw.githubusercontent.com/Cospamos/wifi-cursor/master/install.sh | bash
```

Скрипт скачивает бинарник в `~/.wifi-cursor/bin` и добавляет глобальный
алиас `wifi-cursor` в `~/.bashrc`/`~/.zshrc` (в зависимости от `$SHELL`).
После установки откройте новый терминал (или `source ~/.bashrc`) — и
`wifi-cursor` будет доступен из любой директории.

## Сборка из исходников

Понадобится Go 1.22+.

### Windows

Компилятор C не нужен — весь ввод/вывод мыши реализован через
`user32.dll`/`syscall` на чистом Go, без cgo.

```
go mod tidy   # подтянет golang.org/x/net (используется discovery) — нужен интернет один раз
go build -o wifi-cursor.exe .
```

### Linux

Захват и эмуляция мыши используют X11 (пакеты `robotgo` + `gohook`), поэтому
нужен компилятор C и заголовки X11:

```
sudo apt install gcc libc6-dev libx11-dev xorg-dev libxtst-dev libxkbcommon-dev \
  libxkbcommon-x11-dev libpng-dev libx11-xcb-dev libxcb1-dev libxcb-xkb-dev
go mod tidy   # подтянет robotgo и gohook — нужен интернет один раз
go build -o wifi-cursor .
```

Именно этот набор пакетов проверен и используется в CI (`.github/workflows/ci.yml`) —
если чего-то не хватит на другом дистрибутиве, ориентируйтесь на сообщения
компилятора о недостающих заголовках (`X11/*.h`, `xkbcommon/*.h`).

Работает поверх X11/Xorg (в том числе XWayland). Чистый Wayland-сеанс без
XWayland не поддерживается — у XTest там нет доступа.

## Использование

На первом устройстве:

```
wifi-cursor create
```

Выведет 6-значный ID пула, например `K4RT9X`.

На втором (и последующих) устройствах в той же Wi-Fi сети:

```
wifi-cursor join K4RT9X
```

Если запустить `wifi-cursor join` без ID, утилита 2 секунды слушает сеть и
покажет уже найденные пулы, после чего попросит ввести ID.

Пока утилита запущена:

- **Край экрана** — доведите курсор до левого или правого края, и управление
  перейдёт соседнему устройству (порядок соседей — по кругу, определяется
  одинаково на всех устройствах, без согласования).
- **Ctrl+Alt+1..9** — мгновенно передать курсор устройству №N (по тому же
  круговому порядку), не перетаскивая курсор через промежуточные экраны.
- **Ctrl+C** — выйти из пула.

## Как это устроено

- `internal/protocol` — формат сообщений (JSON по TCP между участниками пула,
  UDP-multicast для обнаружения в сети).
- `internal/discovery` — обнаружение устройств в текущей Wi-Fi сети через
  UDP-multicast (`239.255.42.99:47990`): периодические presence-маячки и
  прямой поиск "у кого есть пул с ID X". Сетевой адаптер для multicast
  выбирается явно (`LocalInterface`), а не отдаётся на откуп ОС — иначе на
  машине с несколькими адаптерами (VPN, Hyper-V, WSL, Docker Desktop)
  система может тихо выбрать не тот интерфейс, и обнаружение перестаёт
  работать между устройствами, хотя оба реально в одном Wi-Fi.
- `internal/pool` — децентрализованное членство пула: полносвязная TCP-сеть
  между участниками, состав поддерживается gossip'ом (без координатора),
  отказы обнаруживаются heartbeat'ом. Право "владеть курсором" передаётся
  по схеме last-writer-wins с монотонной версией; если активное устройство
  пропадает, оставшиеся участники детерминированно выбирают следующее
  (наименьший ID) — без голосования и без единой точки отказа.
- `internal/input` — платформенный захват/эмуляция ввода
  (`input_windows.go` — низкоуровневый хук `WH_MOUSE_LL`/`WH_KEYBOARD_LL` и
  `SendInput`, чистый Go без cgo; `input_linux.go` — X11 через `robotgo` и
  `gohook`).
- `internal/cursor` — состояние "у кого сейчас курсор": определение края
  экрана, обработка горячей клавиши, приём/передача сырых событий мыши от
  неактивного устройства к активному (курсор на неактивном устройстве
  постоянно возвращается в центр экрана, чтобы можно было толкать мышь сколь
  угодно долго — это и позволяет управлять соседним экраном).

## Тестирование discovery/пула в Docker

`testing/docker` поднимает два независимых контейнера ("device-a",
"device-b") в одной bridge-сети — рабочая модель "двух устройств в одной
Wi-Fi" для проверки обнаружения и подключения к пулу без реального железа
(GUI/захват курсора внутри не тестируется, только сеть):

```bash
cd testing/docker
./run-test.sh
```

Скрипт собирает образы, стартует `wifi-cursor create` на device-a, затем
`wifi-cursor join <ID>` на device-b, и печатает `SUCCESS`/`FAILURE`.

## Если create/join не находят друг друга

Протокол обнаружения проверен в изолированном тестовом стенде
(`testing/docker` — два Docker-контейнера в одной bridge-сети как модель
"двух устройств в одной Wi-Fi") — `create` → `join` там отрабатывают с
первого раза. Значит, если у вас `join` не находит пул, дело почти всегда в
сети/файрволе конкретной машины, а не в коде:

1. **Брандмауэр Windows.** По умолчанию блокирует входящие подключения к
   непроверенным `.exe` — из-за этого discovery-пакеты просто не доходят.
   Добавьте разрешающее правило (PowerShell от имени администратора):
   ```powershell
   netsh advfirewall firewall add rule name="wifi-cursor" dir=in action=allow program="ПУТЬ\wifi-cursor.exe" enable=yes profile=any
   ```
2. **Профиль сети "Общедоступная" (Public) в Windows.** Для него правила
   брандмауэра строже по умолчанию. Параметры → Сеть и Интернет → Wi-Fi →
   свойства сети → тип профиля → "Частная".
3. **Изоляция клиентов (AP/client isolation) на роутере.** Часто включена на
   гостевых Wi-Fi-сетях и части mesh-систем по умолчанию — блокирует
   вообще любой трафик между устройствами на одном Wi-Fi, включая broadcast
   и multicast. Если оба устройства на гостевой сети или изоляция включена
   в настройках роутера — обнаружение не заработает в принципе, это не
   лечится на стороне приложения. Проверьте, что оба устройства в одной
   (не гостевой) сети и изоляция клиентов выключена.
4. **Linux-фаервол** (`ufw`/`firewalld`), если включён — аналогично
   потребуется разрешить UDP `47990` и TCP-порт, который печатает
   `wifi-cursor` (или разрешить приложению целиком).

## Известные ограничения прототипа

- Модель "один физический курсор": подразумевается, что мышь физически
  подключена к одному устройству за раз, а остальные лишь показывают/пробрасывают
  ввод. Если реально трогать мышь на нескольких машинах одновременно,
  поведение не определено.
- На Linux нет скрытия/захвата системного курсора (в отличие от Windows,
  где он прячется через `ClipCursor`/`ShowCursor`) — на неактивном устройстве
  курсор будет заметно дёргаться в центр экрана.
- Учитывается только основной монитор каждого устройства.
- Обнаружение работает через UDP-multicast в пределах одной подсети/Wi-Fi;
  сети с изолированными клиентами (client isolation) или сложным NAT не
  поддерживаются.
- Если на устройстве одновременно активно несколько интерфейсов с реальным
  IPv4 (например, и Wi-Fi, и Ethernet), `LocalInterface` берёт первый
  подходящий из списка ОС — не обязательно тот, что реально смотрит в
  нужную сеть. В этом случае помогает временно отключить лишний адаптер.
- Секретность пула держится только на случайности 6-символьного ID —
  шифрования или аутентификации соединений нет; не использовать в
  недоверенных сетях.
- Сборка и `go vet` автоматически проверяются в CI на `windows-latest` и
  `ubuntu-latest` при каждом пуше (см. бейдж выше и `.github/workflows/ci.yml`) —
  так и был найден и исправлен реальный баг (`DX`/`DY` в `protocol.InputEvent`
  делили один JSON-тег, из-за чего дельта движения мыши по Y никогда не
  доходила бы по сети).
